               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Guys I've still been seeing code committed that doesn't match standard Java style or a reasonable policy I can imagine. I wanted to talk about it since I've just been silently changing it and that is not ideal.</p> 
                  <p>This should be easy to get right, as automated tools exist to check and fix this. I recommend IntelliJ's free Community edition. Flip on even basic inspections. A hundred things will jump out (that are already jumping out at me). Most are automatically fixable. </p> 
                  <p>I think that standardized, readable code invites attention, work and care: it feels like something you want to improve, and don't want to hack up.</p> 
                  <p>I think it helps attract committers. Strong engineering organizations wouldn't let basic style problems in the codebase, just by using automated checks. Code reviews don't begin otherwise, and then reviews focus on real issues like design. We can make a basic effort to approach that level of quality. Otherwise, people who are used to a higher standard won't be inclined to participate in the project, and will just fork.</p> 
                  <p>I think it's a prerequisite to fixing real design issues, TODOs, correctness problems (cloning for instance), and refactorings. This code is not near that point, and won't get there at this rate. </p> 
                  <p>Personally it makes we want to only support anything I've written, and write any "next generation" recommender system in a new and separate venture. And I'm a friendly, and maybe not the only one! So would be great to keep some focus on quality and design.</p> 
                  <p>Here's a patch showing all the changes I've picked up and made with the IDE – <b>just</b> basic style issues, and just since the last 2 weeks. The issues are, among others:</p> 
                  <p> ⁃ Empty javadoc<br> ⁃ Redundant javadoc ("@param foo the foo")<br> ⁃ Missing copyright headers<br> ⁃ Copyright headers not at top of file (sometimes after imports!)<br> ⁃ Very long lines (&gt;&gt; 120 chars)<br> ⁃ "throws Exception" not on main() or test method<br> ⁃ "transient" fields – should never be used for us<br> ⁃ Missing @Override<br> ⁃ Using new Random()<br> ⁃ Redundant boolean expressions like "foo == true"<br> ⁃ Unused variables and parameters<br> ⁃ Unused imports<br> ⁃ Loops and conditionals without braces<br> ⁃ Weird literals ("1d")</p> 
                 </div> 
                </div>
