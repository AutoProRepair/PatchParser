                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>When there are multiple constraints on a single table and the constraints have the same set of columns (in the same order), Derby tries to optimize things by re-using a single backing index for all of the relevant constraints. See the "executeConstantAction()" method of CreateIndexConstantAction.java (search for "duplicate").</p> 
                  <p>But there is a bug in Derby where, if one of the constraints is unique and is dropped, the uniqueness "attribute" of the backing index is not updated accordingly. This means that uniqueness may be incorrectly enforced where it is not required.</p> 
                  <p>Take the following example ("Case 2" from <a href="https://issues.apache.org/jira/browse/DERBY-2204" title="Foreign key constraint backing index creation can be smarter when foreign key is a subset of the table's primary key" class="issue-link" data-issue-key="DERBY-2204">DERBY-2204</a>):</p> 
                  <p> ALTER TABLE NEWORDERS ADD CONSTRAINT<br> NEWORDERS_PK PRIMARY KEY(NO_W_ID, NO_D_ID, NO_O_ID);</p> 
                  <p> ALTER TABLE NEWORDERS ADD CONSTRAINT<br> NO_O_FK FOREIGN KEY (NO_W_ID, NO_D_ID, NO_O_ID) REFERENCES ORDERS;</p> 
                  <p>For these statements Derby will use a single backing index for both the primary constraint NEWORDERS_PK and the foreign key constraint NO_O_FK. That backing index will be unique because the primary key must itself be unique.</p> 
                  <p>If later we drop the primary key:</p> 
                  <p> ALTER TABLE NEWORDERS DROP CONSTRAINT NEWORDERS_PK;</p> 
                  <p>then the backing index needs to be converted from a unique index to a non-unique index (because a foreign key is not inherently unique). But in Derby the uniqueness attribute remains unchanged, so attempts to insert a duplicate (NO_W_ID, NO_D_ID, NO_O_ID) row into NEWORDERS will fail with error 23505, when it should really succeed.</p> 
                  <p>I tried this out on 10.1.3.1 and the same behavior occurs there, so marking "Affects" versions for everything back to that...</p> 
                 </div> 
                </div>
