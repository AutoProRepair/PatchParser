                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>While trying out some code changes for <a href="https://issues.apache.org/jira/browse/DERBY-47" title="Some possible improvements to IN optimization" class="issue-link" data-issue-key="DERBY-47"><del>DERBY-47</del></a> I was running a few test cases and happened to notice that there are a couple of cases in which Derby behaves incorrectly (that or my understanding of what should be happening here is way off).</p> 
                  <p>First and most simply: the following query should return zero rows (unless I'm missing something?), but it returns one:</p> 
                  <p>ij&gt; create table t1 (i int);<br> 0 rows inserted/updated/deleted<br> ij&gt; insert into t1 values 1, 2, 3, 4, 5;<br> 5 rows inserted/updated/deleted</p> 
                  <p>– Correct returns zero rows.<br> ij&gt; select * from t1 where i in (4.23);<br> I<br> -----------</p> 
                  <p>0 rows selected</p> 
                  <p>– But this one returns 1 row...<br> ij&gt; select * from t1 where i in (2.8, 4.23);<br> I<br> -----------<br> 4</p> 
                  <p>1 row selected</p> 
                  <p>Secondly, if the IN-list contains a non-constant value then Derby can incorrectly return rows that do not match the IN predicate. I think this is because some internal casting is happening when it shouldn't?</p> 
                  <p>ij&gt; create table t1 (i int);<br> 0 rows inserted/updated/deleted<br> ij&gt; insert into t1 values 1, 2, 3, 4, 5;<br> 5 rows inserted/updated/deleted</p> 
                  <p>– Following values clause returns "2.80", as expected.<br> ij&gt; values cast (2.8 as decimal(4, 2));<br> 1<br> -------<br> 2.80</p> 
                  <p>1 row selected</p> 
                  <p>– But if we use it in an IN-list it gets cast to "2" and thus returns a match.<br> – We get 2 rows when we should get NONE.<br> ij&gt; select * from t1 where i in (cast (2.8 as decimal(4, 2)), 4.23);<br> I<br> -----------<br> 2<br> 4</p> 
                  <p>2 rows selected</p> 
                  <p>I confirmed that we see these results on trunk, 10.2, 10.1, and even as far back as svn #201660 for 10.0. I also ran the above statements on DB2 v8 as a sanity check to confirm that NO results were returned there.</p> 
                 </div>
