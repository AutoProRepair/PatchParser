                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>When rejoining a global transaction, client does not report the correct isolation level with a <br> getTransactionIsolation(). The server side isolation should be ok I think.</p> 
                  <p>This was discovered when testing the fix for <a href="https://issues.apache.org/jira/browse/DERBY-1044" title="[xa] client XAConnection.getConnection()  does not have the correct  default isolation level if set by an earlier connection obtained from the same XAConnection" class="issue-link" data-issue-key="DERBY-1044"><del>DERBY-1044</del></a>. After the fix for <a href="https://issues.apache.org/jira/browse/DERBY-1044" title="[xa] client XAConnection.getConnection()  does not have the correct  default isolation level if set by an earlier connection obtained from the same XAConnection" class="issue-link" data-issue-key="DERBY-1044"><del>DERBY-1044</del></a>, there is a new diff in the test, but the fix for <a href="https://issues.apache.org/jira/browse/DERBY-1044" title="[xa] client XAConnection.getConnection()  does not have the correct  default isolation level if set by an earlier connection obtained from the same XAConnection" class="issue-link" data-issue-key="DERBY-1044"><del>DERBY-1044</del></a> just exposed this issue. The output for the test was correct before by circumstance.</p> 
                  <p>I will put comments with this bug in checkDataSource test.</p> 
                  <p>// now re-join the transaction, should pick up the read-only<br> // and isolation level from the transaction,<br> // holdability remains that of this handle.<br> xar.start(xid, XAResource.TMJOIN);<br> printState("re-join X1", cs1);<br> xar.end(xid, XAResource.TMSUCCESS);</p> 
                 </div> 
                </div>
