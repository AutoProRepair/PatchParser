                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The classes in package "analysis.solvers" could be refactored similarly to what was done for package <tt>optimization</tt>.</p> 
                  <ul> 
                   <li>Replace <tt>MaxIterationsExceededException</tt> with <tt>TooManyEvaluationsException</tt>:<br> Apart from the class <tt>MaxIterationsExceededException</tt> being deprecated, this approach makes it difficult to compare different algorithms: While the concept of iteration is algorithm-dependent, the user is probably mostly interested in the number of function evaluations. </li> 
                   <li>Implement the method <tt>solve</tt> in the base class (<tt>UnivariateRealSolverImpl</tt>) and define an abstract method <tt>doSolve</tt> to be implemented in derived classes. This method would then use a new <tt>computeObjectiveFunction</tt> method that will take care of the counting of the function evaluations.</li> 
                   <li>Remove "protected" fields (the root is unnecessary since it is returned by <tt>solve</tt>). Arguingly the function value is also not very useful (as we know what it should be), except for debugging purposes (in which case, it might not be a problem to call the function's <tt>value</tt> method once more).</li> 
                   <li>Remove the tolerance setter (accuracy) and make the corresponding fields "final".</li> 
                  </ul> 
                 </div>
