                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>While running our app overnight, performing some intensive database operations (primarily deletes), we experienced a transaction failure, with error message "A SAVEPOINT with the passed name already exists in the current transaction". This failure appears to be very intermittent, as I've run the same operation successfully for hours and hours without failure. Some more information:</p> 
                  <p>1. I see that there are four JIRA issues which report the same message (<a href="https://issues.apache.org/jira/browse/DERBY-2773" title=" testGrantRevokeDDL(org.apache.derbyTesting.functionTests.tests.lang.GrantRevokeDDLTest) throws java.sql.SQLException: The exception 'java.sql.SQLException: A SAVEPOINT with the passed name already exists in the current transaction.' " class="issue-link" data-issue-key="DERBY-2773"><del>DERBY-2773</del></a>, <a href="https://issues.apache.org/jira/browse/DERBY-1457" title="lang/triggerGeneral.sql fails intermittently" class="issue-link" data-issue-key="DERBY-1457"><del>DERBY-1457</del></a>, <a href="https://issues.apache.org/jira/browse/DERBY-2808" title="triggerRecursion.sql and triggerGeneral.sql intermittently fail with SAVEPOINT error." class="issue-link" data-issue-key="DERBY-2808"><del>DERBY-2808</del></a>, <a href="https://issues.apache.org/jira/browse/DERBY-2804" title="predicatesIntoViews.sql fails (intermittently?) with SAVEPOINT error w/ IBM 15 on Linux." class="issue-link" data-issue-key="DERBY-2804"><del>DERBY-2804</del></a>). I am not sure to what extent if any, the problem I experienced is related to these issues.</p> 
                  <p>• At least three of the above JIRA issues are related to triggers. Our transaction failure occurred while performing a delete in a table which has delete triggers.<br> • We aren't using savepoints explicitly. Of course, it can be seen in the stack trace that the trigger operation is setting a savepoint.<br> • One of the JIRA issues mentions exceeding maximum depth of nested triggers. Not sure what a nested trigger is, but the trigger we are performing should not (in theory) cause another trigger event.</p> 
                  <p>2. Here's the stack trace:</p>

                  <p>3. Here are the relevant tables and triggers. We have different tables for different "sample" data types in our application (currently int, long, float, double, Boolean, string). I have shown the int_sample table; the other tables are identical but have a different data type for the value field. </p> 
                  <p>Sample records are first class; time records are second class - they only exist to support sample records. When there are no remaining sample records for a given time record, the time record can be removed. It is the job of the trigger to perform this cleanup: when a record is deleted from any of the sample tables, the time record corresponding to *_sample.fk_time_id is deleted only if there are no remaining records in any of the sample tables which have the same fk_time_id. So although there are cascading deletes (deleting a time record deletes all of its samples), the trigger should not recurse because the trigger only deletes when there are no associated records.</p> 
                  <p>create table time (<br> id int not null generated always as identity,<br> time timestamp not null,</p> 
                  <p> constraint time_pk primary key (id),<br> constraint time_unique unique (time)<br> );</p> 
                  <p>create table int_sample (<br> fk_band_id int not null,<br> fk_time_id int not null,<br> value int not null,</p> 
                  <p> constraint int_sample_pk primary key (fk_band_id, fk_time_id),<br> constraint int_sample_fk_band foreign key (fk_band_id) references band (id) on delete cascade,<br> constraint int_sample_fk_time foreign key (fk_time_id) references time (id) on delete cascade<br> );</p> 
                  <p>create trigger cascade_delete_unused_time_for_int_sample_trigger<br> after delete on int_sample<br> referencing old as oldrow<br> for each row<br> delete from time where <br> time.id = oldrow.fk_time_id<br> and id not in (select fk_time_id from int_sample) <br> and id not in (select fk_time_id from long_sample) <br> and id not in (select fk_time_id from float_sample) <br> and id not in (select fk_time_id from double_sample) <br> and id not in (select fk_time_id from boolean_sample) <br> and id not in (select fk_time_id from string_sample);</p> 
                  <p>4. Invocation. This is very rough pseudocode focusing on the core logic. In reality, there's a lot more code, but I'm doing the usual stuff, such as using prepared statements, and closing all resources when done. The key thing is that I'm using two statements in a nested fashion; in the outer statement, I iterate over records in the time table, and in the inner loop I use a second statement to delete records in one of the sample tables. I can provide the actual code if desired.</p> 
                  <p>deleteSamples(int bandId, String sampleTable, Object someQueryCriteria) {<br> within transaction {<br> create stmt1 iterating over time ids in time table (restricted by someQueryCriteria)<br> for (time_id in stmt1) </p> { PreparedStatement stmt2 = conn.prepareStatement(String.format( "delete from %s where fk_band_id = ? and fk_time_id = ?", sampleTable)); set stmt2 parameters and executeUpdate() } 
                  <p> }<br> }</p>
