                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Looking the scorers for PhraseQuery, I think there are some speedups<br> we could do:</p> 
                  <ul> 
                   <li>The AND part of the scorer (which advances to the next doc that<br> has all the terms), in PhraseScorer.doNext, should do the same<br> optimizing as BooleanQuery's ConjunctionScorer, ie sort terms from<br> rarest to most frequent. I don't think it should use a linked<br> list/firstToLast() that it does today.</li> 
                  </ul> 
                  <ul> 
                   <li>We do way too much work now when .score() is not called, because<br> we go and find all occurrences of the phrase in the doc, whereas<br> we should stop only after finding the first and then go and count<br> the rest if .score() is called.</li> 
                  </ul> 
                  <ul> 
                   <li>For the exact case, I think we can use two int arrays to find the<br> matches. The first array holds the count of how many times a term<br> in the phrase "matched" a phrase starting at that position. When<br> that count == the number of terms in the phrase, it's a match.<br> The 2nd is a "gen" array (holds docID when that count was last<br> touched), to avoid clearing. Ie when incrementing the count, if<br> the docID != gen, we reset count to 0. I think this'd be faster<br> than the PQ we now use. Downside of this is if you have immense<br> docs (position gets very large) we'd need 2 immense arrays.</li> 
                  </ul> 
                  <p>It'd be great to do <a href="https://issues.apache.org/jira/browse/LUCENE-1252" title="Avoid using positions when not all required terms are present" class="issue-link" data-issue-key="LUCENE-1252"><del>LUCENE-1252</del></a> along with this, ie factor<br> PhraseScorer into two AND'd sub-scorers (<a href="https://issues.apache.org/jira/browse/LUCENE-1252" title="Avoid using positions when not all required terms are present" class="issue-link" data-issue-key="LUCENE-1252"><del>LUCENE-1252</del></a> is open for<br> this). The first one should be ConjunctionScorer, and the 2nd one<br> checks the positions (ie, either the exact or sloppy scorers). This<br> would mean if the PhraseQuery is AND'd w/ other clauses (or, a filter<br> is applied) we would save CPU by not checking the positions for a doc<br> unless all other AND'd clauses accepted the doc.</p> 
                 </div> 
                </div> 
               </div>
