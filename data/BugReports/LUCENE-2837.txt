               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>We've discussed cleaning up our *Searcher stack for some time... I<br> think we should try to do this before releasing 4.0.</p> 
                  <p>So I'm attaching an initial patch which:</p> 
                  <ul> 
                   <li>Removes Searcher, Searchable, absorbing all their methods into IndexSearcher</li> 
                  </ul> 
                  <ul> 
                   <li>Removes contrib/remote</li> 
                  </ul> 
                  <ul> 
                   <li>Removes MultiSearcher</li> 
                  </ul> 
                  <ul> 
                   <li>Absorbs ParallelMultiSearcher into IndexSearcher (ie you can now<br> pass useThreads=true, or a custom ES to the ctor)</li> 
                  </ul> 
                  <p>The patch is rough – I just ripped stuff out, did search/replace to<br> IndexSearcher, etc. EG nothing is directly testing using threads with<br> IndexSearcher, but before committing I think we should add a<br> newSearcher to LuceneTestCase, which randomly chooses whether the<br> searcher uses threads, and cutover tests to use this instead of making<br> their own IndexSearcher.</p> 
                  <p>I think MultiSearcher has a useful purpose, but as it is today it's<br> too low-level, eg it shouldn't be involved in rewriting queries: the<br> Query.combine method is scary. Maybe in its place we make a higher<br> level class, with limited API, that's able to federate search across<br> multiple IndexSearchers? It'd also be able to optionally use thread<br> per IndexSearcher.</p> 
                 </div>
