                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The bug is illustrated by the following code snippet:</p> 
                  <div class="code panel" style="border-width: 1px;">
                   <div class="codeContent panelContent"> 
                    <pre class="code-java">        Connection c = DriverManager.getConnection(<span class="code-quote">"jdbc:derby:memory:db;create=<span class="code-keyword">true</span>"</span>);
        c.setAutoCommit(<span class="code-keyword">false</span>);
        Statement s = c.createStatement();
        s.execute(<span class="code-quote">"create table t1(x <span class="code-object">int</span> primary key initially deferred)"</span>);
        s.execute(<span class="code-quote">"insert into t1 values 1,1,1,1"</span>);

        Savepoint sp = c.setSavepoint();
        s.execute(<span class="code-quote">"drop table t1"</span>);
        c.rollback(sp);

        <span class="code-comment">// Since there are four identical rows in T1, <span class="code-keyword">this</span> call should have
</span>        <span class="code-comment">// failed because the primary key was violated.
</span>        c.commit();

        <span class="code-comment">// Instead, it succeeds, and all four rows are committed, as can
</span>        <span class="code-comment">// be seen here:
</span>        ResultSet rs = s.executeQuery(<span class="code-quote">"select * from t1"</span>);
        <span class="code-keyword">while</span> (rs.next()) {
            <span class="code-object">System</span>.out.println(rs.getInt(1));
        }

        <span class="code-comment">// Insert yet another row, so that we have five identical rows ...
</span>        s.execute(<span class="code-quote">"insert into t1 values 1"</span>);
        <span class="code-comment">// ... and now commit complains ...
</span>        c.commit();
</pre> 
                   </div>
                  </div> 
                  <p>With auto-commit off, add duplicates into a deferred primary key. Then set a savepoint, drop the table, and roll back to the savepoint.</p> 
                  <p>Apparently, when you drop the table, information about any constraint violations seen on that table is lost, and that information is not restored when the drop table operation is undone by the rollback to savepoint.</p> 
                  <p>So when you commit the transaction after having rolled back the drop operation, no deferred checking of constraints happens, and the duplicates you have inserted are committed.</p> 
                 </div> 
                </div> 
               </div>
