                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>I've encountered what appears to be a bug related to threading. After an INSERT operation, if the invoking thread terminates too quickly, Derby throws an XSDG.</p> 
                  <p>The bug is a bit difficult to isolate but it occurs consistently in the following situation (with a particular database and an operation of a particular size):</p> 
                  <p>Derby is running in embedded mode with autocommit on. <br> The application performs an INPUT operation from a thread that is not the main thread. The INPUT is issued using a PreparedStatement. The INPUT adds ~ 256 records of six fields each. (Note that INSERTs of this size seem to work fine in other contexts.)</p> 
                  <p>The preparedStatement.executeUpdate() seems to excute successfully; at least it returns without throwing an exception. </p> 
                  <p>The thread that invoked the INPUT operation then terminates (but NOT the application). The next INPUT operation then results in an</p> 
                  <p>"ERROR XSDG1: Page Page(7,Container(0, 1344)) could not be written to disk, please check if disk is full."</p> 
                  <p>The disk is definitely not full.</p> 
                  <p>HOWEVER, if I put the calling thread to sleep for a second before it exits, the problem does not occur.</p> 
                  <p>I'm not quite sure what to make of this. I was under the impression that most of Derby's activity occurs in the application's threads. Could Derby be creating a child thread from in the application thread, which dies when the parent thread terminates?</p> 
                  <p>Thanks</p> 
                 </div> 
                </div> 
               </div>
