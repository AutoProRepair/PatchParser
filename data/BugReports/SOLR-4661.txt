                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>the ReplicationHandler (and the replication admin UI screen) report the index version &amp; gen for the master based on what commit point is currently open for searching – but this is not necessarily the most recent commit point available for replication.</p> 
                  <p>Thus, it can appear that a slave has "gotten ahead" of the master, if there are "empty commits" (because of reader reopening shotcuts) or commits using openSearcher=false.</p> 
                  <p>We need to add additional data to help make it clear there is no actual problem in this sitation.</p> 
                  <p>Summary of original bug report..</p> 
                  <div class="panel" style="border-width: 1px;">
                   <div class="panelContent"> 
                    <p>Index and Gen number on Slave is higher than master. <br> If you apply commit on master with no pending docs then the commit time stamp and gen is incremented. When Slaves polls master for replication it see the index version difference and starts replicating but all files are skipped. </p> 
                    <p>On Admin UI (on Slaves) the version number displayed for master is old where as for slave is the latest which is higher than master.<br> Below is the response from master (/replication?command=details) where i see two different Version an Gen numbers. This creates confusion of having version out of sync, though its not. </p> 
                    <p>...</p> 
                   </div>
                  </div>
