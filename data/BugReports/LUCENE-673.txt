                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>I'm opening this issue to track details on the known problems with<br> Lucene over NFS.</p> 
                  <p>The summary is: if you have one machine writing to an index stored on<br> an NFS mount, and other machine(s) reading (and periodically<br> re-opening the index) then sometimes on re-opening the index the<br> reader will hit a FileNotFound exception.</p> 
                  <p>This has hit many users because this is a natural way to "scale up"<br> your searching (single writer, multiple readers) across machines. The<br> best current workaround (I think?) is to take the approach Solr takes<br> (either by actually using Solr or copying/modifying its approach) to<br> take snapshots of the index and then have the readers open the<br> snapshots instead of the "live" index being written to.</p> 
                  <p>I've been working on two patches for Lucene:</p> 
                  <ul> 
                   <li>A locking (LockFactory) implementation using native OS locks</li> 
                  </ul> 
                  <ul> 
                   <li>Lock-less commits</li> 
                  </ul> 
                  <p>(I'll open separate issues with the details for those).</p> 
                  <p>I have a simple stress test where one machine is constantly adding<br> docs to an index over NFS, and another machine is constantly<br> re-opening the index searcher over NFS.</p> 
                  <p>These tests have revealed new details (at least for me!) about the<br> root cause of our NFS problems:</p> 
                  <ul> 
                   <li>Even when using native locks over NFS, Lucene still hits these<br> exceptions!</li> 
                  </ul> 
                  <p> I was surprised by this because I had always thought (assumed?)<br> the NFS problem was because the "simple" file-based locking was<br> not correct over NFS, and that switching to native OS filesystem<br> locking would resolve it, but it doesn't.</p> 
                  <p> I can reproduce the "FileNotFound" exceptions even when using NFS<br> V4 (the latest NFS protocol), so this is not just a "your NFS<br> server is too old" issue.</p> 
                  <ul> 
                   <li>Then, when running the same stress test with the lock-less<br> changes, I don't hit any exceptions. I've tested on NFS version<br> 2, 3 and 4 (using the "nfsvers=N" mount option).</li> 
                  </ul> 
                  <p>I think this means that in fact (as Hoss at one point suggested I<br> believe), the NFS problems are likely due to the cache coherence of<br> the NFS file system (I think the "segments" file in particular)<br> against the existence of the actual segment data files.</p> 
                  <p>In other words, even if you lock correctly, on the reader side it will<br> sometimes see stale contents of the "segments" file which lead it to<br> try to open a now deleted segment data file.</p> 
                  <p>So I think this is good news / bad news: the bad news is, native<br> locking doesn't fix our problems with NFS (as at least I had expected<br> it to). But the good news is, it looks like (still need to do more<br> thorough testing of this) the changes for lock-less commits do enable<br> Lucene to work fine over NFS.</p> 
                  <p>[One quick side note in case it helps others: to get native locks<br> working over NFS on Ubuntu/Debian Linux 6.06, I had to "apt-get<br> install nfs-common" on the NFS client machines. Before I did this I<br> would hit "No locks available" IOExceptions on calling the "tryLock"<br> method. The default nfs server install on the server machine just<br> worked because it runs in kernel mode and it start a lockd process.]</p> 
                 </div> 
                </div>
