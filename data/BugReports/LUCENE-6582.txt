               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Some time ago, I had a problem with synonyms and phrase type queries (actually, it was elasticsearch and I was using a match query with multiple terms and the "and" operator, as better explained here: <a href="https://github.com/elastic/elasticsearch/issues/10394" class="external-link" rel="nofollow">https://github.com/elastic/elasticsearch/issues/10394</a>).</p> 
                  <p>That issue led to some work on Lucene: <a href="https://issues.apache.org/jira/browse/LUCENE-6400" title="SynonymParser should encode 'expand' correctly." class="issue-link" data-issue-key="LUCENE-6400"><del>LUCENE-6400</del></a> (where I helped a little with tests) and <a href="https://issues.apache.org/jira/browse/LUCENE-6401" title="Refactor hairy logic of QueryBuilder" class="issue-link" data-issue-key="LUCENE-6401"><del>LUCENE-6401</del></a>. This issue is also related to <a href="https://issues.apache.org/jira/browse/LUCENE-3843" title="implement PositionLengthAttribute for all tokenstreams where its appropriate" class="issue-link" data-issue-key="LUCENE-3843">LUCENE-3843</a>.</p> 
                  <p>Starting from the discussion on <a href="https://issues.apache.org/jira/browse/LUCENE-6400" title="SynonymParser should encode 'expand' correctly." class="issue-link" data-issue-key="LUCENE-6400"><del>LUCENE-6400</del></a>, I'm attempting to implement a solution. Here is a patch with a first step - the implementation to fix "SynFilter to be able to 'make positions'" (as was mentioned on the <a href="https://issues.apache.org/jira/browse/LUCENE-6400?focusedCommentId=14498554&amp;page=com.atlassian.jira.plugin.system.issuetabpanels:comment-tabpanel#comment-14498554" class="external-link" rel="nofollow">issue</a>). In this way, the synonym filter generates a correct (or, at least, better) graph.</p> 
                  <p>As the synonym matching is greedy, I only had to worry about fixing the position length of the rules of the current match, no future or past synonyms would "span" over this match (please correct me if I'm wrong!). It did require more buffering, twice as much.</p> 
                  <p>The new behavior I added is not active by default, a new parameter has to be passed in a new constructor for <tt>SynonymFilter</tt>. The changes I made do change the token stream generated by the synonym filter, and I thought it would be better to let that be a voluntary decision for now.</p> 
                  <p>I did some refactoring on the code, but mostly on what I had to change for may implementation, so that the patch was not too hard to read. I created specific unit tests for the new implementation (<tt>TestMultiWordSynonymFilter</tt>) that should show how things will be with the new behavior.</p> 
                 </div>
