                <h2 class="toggle-title">Description</h2>

               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p><font color="red">The work on this issue is temporarily at github</font> (lots of experiments and tweaking):<br> <a href="https://github.com/carrotsearch/randomizedtesting" class="external-link" rel="nofollow">https://github.com/carrotsearch/randomizedtesting</a><br> Or directly: git clone git://github.com/carrotsearch/randomizedtesting.git<br> <font color=""></font></p> 
                  <hr> 
                  <p>RandomizedRunner is a JUnit runner, so it is capable of running @Test-annotated test cases. It<br> respects regular lifecycle hooks such as @Before, @After, @BeforeClass or @AfterClass, but it<br> also adds the following:</p> 
                  <p>Randomized, but repeatable execution and infrastructure for dealing with randomness:</p> 
                  <ul class="alternate" type="square"> 
                   <li>uses pseudo-randomness (so that a given run can be repeated if given the same starting seed)<br> for many things called "random" below,</li> 
                   <li>randomly shuffles test methods to ensure they don't depend on each other,</li> 
                   <li>randomly shuffles hooks (within a given class) to ensure they don't depend on each other,</li> 
                   <li>base class RandomizedTest provides a number of methods for generating random numbers, strings<br> and picking random objects from collections (again, this is fully repeatable given the<br> initial seed if there are no race conditions),</li> 
                   <li>the runner provides infrastructure to augment stack traces with information about the initial<br> seeds used for running the test, so that it can be repeated (or it can be determined that<br> the test is not repeatable – this indicates a problem with the test case itself).</li> 
                  </ul> 
                  <p>Thread control:</p> 
                  <ul class="alternate" type="square"> 
                   <li>any threads created as part of a test case are assigned the same initial random seed<br> (repeatability),</li> 
                   <li>tracks and attempts to terminate any Threads that are created and not terminated inside<br> a test case (not cleaning up causes a test failure),</li> 
                   <li>tracks and attempts to terminate test cases that run for too long (default timeout: 60 seconds,<br> adjustable using global property or annotations),</li> 
                  </ul> 
                  <p>Improved validation and lifecycle support:</p> 
                  <ul class="alternate" type="square"> 
                   <li>RandomizedRunner uses relaxed contracts of hook methods' accessibility (hook methods <em>can</em> be<br> private). This helps in avoiding problems with method shadowing (static hooks) or overrides<br> that require tedious super.() chaining). Private hooks are always executed and don't affect<br> subclasses in any way, period.</li> 
                   <li>@Listeners annotation on a test class allows it to hook into the execution progress and listen<br> to events.</li> 
                   <li>@Validators annotation allows a test class to provide custom validation strategies<br> (project-specific). For example a base class can request specific test case naming strategy<br> (or reject JUnit3-like methods, for instance).</li> 
                   <li>RandomizedRunner does not "chain" or "suppress" exceptions happening during execution of<br> a test case (including hooks). All exceptions are reported as soon as they happened and multiple<br> failure reports can occur. Most environments we know of then display these failures sequentially<br> allowing a clearer understanding of what actually happened first.</li> 
                  </ul> 
                 </div> 
                </div>

If anything, generate per-thread Randoms based on current seed and let each thread handle its own sequence of pseudo-random numbers from there. This is even possible at runtime with non-mock objects as I\'m going to show in Barcelona, hopefully. \\n \\n\\n\\n\\n Now\\u2026 if you\'re still with me you\'re probably interested how this applies to Lucene. The wall I\'ve hit is the sheer amount of code that any change to LTC affects. I realized it\'d be large, but it\'s just gargantuan    \\n\\n The major issue is with static initializers and static public methods called from them that leave resources behind. I\'m sorry, but nobody can convince me this isn\'t evil. I understand certain things are costly and require a one-time setup, but these should really be moved to @BeforeClass fixture hooks. If one really needs to do things once at JVM lifespan level a @BeforeClass with some logic to perform a single initialization can be a replacement for a static initializer (even if it\'s unclear to me when exactly such a fixture would be really needed). In short: the problem with static initializers is that they are executed outside the lifecycle control of the runner\\u2026 I\'d say most of the problems and current patchy solutions inside LTC (dealing with resource tracking for example) are somehow related to the fact that static initializers and static method calls are used throughout the codebase.  \\n\\n I am currently wondering if it\'s feasible to provide a single patch that will make a drop-in replacement of LTC.
