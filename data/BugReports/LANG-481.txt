                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The hashCode() methods of the range classes look very suspicious to me. The value is lazily initialized, but the calculation is done _on the cached value. With some unlucky timing a caller may get an incomplete hash.</p> 
                  <p>An unlucky sequence of Code could be something like<br> T1: if (hashCode == 0) // true<br> T1: hashCode = 17;<br> T2: if (hashCode == 0) // now false because hashCode was already set to 17<br> T2: return hashCode; // return 17<br> T1: hashCode = 37 * hashCode...........</p> 
                  <p>where T1 and T2 are different threads accessing the method in parallel and T2 gets the wrong hash "17".</p> 
                  <p>Affected classes are</p> 
                  <p>org.apache.commons.lang.math.DoubleRange<br> org.apache.commons.lang.math.FloatRange<br> org.apache.commons.lang.math.IntRange<br> org.apache.commons.lang.math.LongRange<br> org.apache.commons.lang.math.NumberRange<br> org.apache.commons.lang.math.Range</p> 
                  <p>Possible fix: calculate the hash on a temporary variable and finally assign it to the member</p> 
                 </div> 
                </div> 
               </div>
              </div>
