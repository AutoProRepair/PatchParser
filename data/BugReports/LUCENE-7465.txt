                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>I think there are some nice benefits to a version of PatternTokenizer that uses Lucene's RegExp impl instead of the JDK's:</p> 
                  <ul> 
                   <li>Lucene's RegExp is compiled to a DFA up front, so if a "too hard" RegExp is attempted the user discovers it up front instead of later on when a "lucky" document arrives</li> 
                  </ul> 
                  <ul> 
                   <li>It processes the incoming characters as a stream, only pulling 128 characters at a time, vs the existing <tt>PatternTokenizer</tt> which currently reads the entire string up front (this has caused heap problems in the past)</li> 
                  </ul> 
                  <ul> 
                   <li>It should be fast.</li> 
                  </ul> 
                  <p>I named it <tt>SimplePatternTokenizer</tt>, and it still needs a factory and improved tests, but I think it's otherwise close.</p> 
                  <p>It currently does not take a <tt>group</tt> parameter because Lucene's RegExps don't yet implement sub group capture. I think we could add that at some point, but it's a bit tricky.</p> 
                  <p>This doesn't even have group=-1 support (like String.split) ... I think if we did that we should maybe name it differently (<tt>SimplePatternSplitTokenizer</tt>?).</p> 
                 </div> 
                </div>
