                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The EmbedBlob.setPosition implementation has two performance problems when the<br> Blob is represented by a store stream, at least one of them rather significant:</p> 
                  <p> 1) The store stream is reset to position zero for each position request.<br> Data is then read until the requested position has been reached.</p> 
                  <p> 2) 'read' is used instead of 'skip', which causes Derby to miss out on the<br> optimization potential with streams that have a more efficient skip mechanism.</p> 
                  <p>My gut feeling is that once point 1) has been fixed, point 2) will have<br> disappeared. Also note that the reason why the unconditional reset approach was<br> chosen was because the blob implementation couldn't keep track of the underlying<br> streams position. This issue still has to be addressed.</p> 
                  <p>Performance degradation<br> =======================<br> Observations suggest the following approximation can be used to quantify the<br> number of bytes that have to be processed on the server. Goes for both the<br> embedded and the client driver when using the EmbedBlob.getBytes call.<br> s = size of the blob<br> b = buffer/block size<br> n = s/b (number of iterations needed to read the whole Blob)</p> 
                  <p> s + b * n * (n+1) / 2 = number of bytes processed on the server side<br> From now on, I ignore the s.</p> 
                  <p>For a 1 MB Blob when using a buffer of 32 000 bytes, we get:<br> n = 1 * 1'024 * 1'024 / 32'000 ~ 33<br> 32'000 * 33 * (33+1) / 2 = 17'952'000 ~ 17 MB</p> 
                  <p>To quickly verify the approximation I summed up the bytes processed by<br> EmbedBlob.getBytes(long,int) and EmbedBlob.setPosition(long) with the 64 MB<br> Blob used by the repro for <a href="https://issues.apache.org/jira/browse/DERBY-550" title="BLOB : java.lang.OutOfMemoryError with network JDBC driver (org.apache.derby.jdbc.ClientDriver)" class="issue-link" data-issue-key="DERBY-550"><del>DERBY-550</del></a> (modified to use 32'000 read buffer):</p> 
                  <ul class="alternate" type="square"> 
                   <li>approximation: 32'000 * 2097 * (2097+1) / 2 = 70392096000 ~ 66 GB</li> 
                   <li>Derby byte count = 70459204864 ~ 66 GB</li> 
                   <li>Derby byte count (buffer 33'000, see below) = 136526134864 ~ 127 GB</li> 
                  </ul> 
                  <p>I'll explain the biggest number further down.<br> As you see, the number of bytes processed is huge. Note that all of the<br> gigabytes are just the 64 MB repeated over and over again. Since the actual data<br> volume is so small, all the data will be in the caches of Derby and the<br> operating system. Note that only 64 MB is actually transferred to the client<br> when using the client driver.</p> 
                  <p>Another consequence of processing all this data repeatedly, is the effect it has<br> on the page cache. Pages has to be evicted and read back in. The performance hit<br> taken by this depends on the page cache size, operating system buffers and other<br> database activities on the system.</p> 
                  <p>The explanation for the largest number above, is another performance issue in<br> the client driver, related to locators. I'll explain it more detailed in a<br> separate Jira issue, but in short the issue causes the stream to be reset even<br> more frequently for read buffer sizes over 32'672 bytes.</p> 
                  <p>Suggested fix<br> =============<br> My initial analysis suggests that the problem can be fixed by using the<br> functionality provided by PosistionedStoreStream. There are a few complicating<br> issues:<br> a) The length encoding bytes must be accounted for properly.<br> b) One must make sure all stream access happens through the<br> PositionedStoreStream, otherwise the position will be incorrect and the<br> wrong data will be returned.</p> 
                  <p>With a prototype fix, the repro duration went down from minutes (7 minutes<br> for the 127 GB case) down to between 2 and 4 seconds with a sane build, running<br> on localhost.</p> 
                  <p>Affected versions<br> =================<br> The code suffering from the performance issues is old, but because it isn't<br> used in the same way in all versions some releases are more affected than<br> others.</p> 
                  <p>Version Embedded.getBytes Client.getXXX<br> 10.5 X X<br> 10.4.1.3 X X<br> 10.3.3.0 X X<br> 10.2.2.1 X _<br> 10.1.3.1 X _<br> (X = issue present)</p> 
                 </div>
