               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Result:<br> After applying a fix for this issue, tests which we conducted show more than 40 percent improvement on our insertion performance.</p> 
                  <p>Explanation:</p> 
                  <p>Using JVM profiler, we found a CPU "bottleneck" during Solr indexing process. This bottleneck can be found at org.apache.solr.schema.IndexSchema, in the following method, "getCopyFieldsList()":</p> 
                  <div class="code panel" style="border-style: solid;border-width: 1px;">
                   <div class="codeHeader panelHeader" style="border-bottom-width: 1px;border-bottom-style: solid;">
                    <b>getCopyFieldsList()</b>
                   </div>
                   <div class="codeContent panelContent"> 
                    <pre class="code-java"><span class="code-keyword">final</span> List&lt;CopyField&gt; result = <span class="code-keyword">new</span> ArrayList&lt;&gt;();
    <span class="code-keyword">for</span> (DynamicCopy dynamicCopy : dynamicCopyFields) {
      <span class="code-keyword">if</span> (dynamicCopy.matches(sourceField)) {
        result.add(<span class="code-keyword">new</span> CopyField(getField(sourceField), dynamicCopy.getTargetField(sourceField), dynamicCopy.maxChars));
      }
    }
    List&lt;CopyField&gt; fixedCopyFields = copyFieldsMap.get(sourceField);
    <span class="code-keyword">if</span> (<span class="code-keyword">null</span> != fixedCopyFields) {
      result.addAll(fixedCopyFields);
    }
</pre> 
                   </div>
                  </div> 
                  <p>This function tries to find for an input source field all its copyFields (All its destinations which Solr need to move this field). <br> As you can probably note, the first part of the procedure is the procedure most “expensive” step (takes O( n ) time while N is the size of the "dynamicCopyFields" group).<br> The next part is just a simple "hash" extraction, which takes O(1) time. </p> 
                  <p>Our schema contains over then 500 copyFields but only 70 of then are "indexed" fields. <br> We also have one dynamic field with a wildcard ( * ), which "catches" the rest of the document fields. <br> As you can conclude, we have more than 400 copyFields that are based on this dynamicField but all, except one, are fixed (i.e. does not contain any wildcard).</p> 
                  <p>From some reason, the copyFields registration procedure defines those 400 fields as "DynamicCopyField " and then store them in the “dynamicCopyFields” array, <br> This step makes getCopyFieldsList() very expensive (in CPU terms) without any justification: All of those 400 copyFields are not glob and therefore do not need any complex pattern matching to the input field. They all can be store at the "fixedCopyFields".<br> Only copyFields with asterisks need this "special" treatment and they are (especially on our case) pretty rare. </p> 
                  <p>Therefore, we created a patch which fix this problem by changing the registerCopyField() procedure.<br> Test which we conducted show that there is no change in the Indexing results. Moreover, the fix still successfully passes the class unit tests (i.e. IndexSchemaTest.java).</p> 
                 </div> 
                </div> 
               </div>
