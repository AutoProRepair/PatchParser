                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>We added a custom field type to allow an indexed binary field type that supports search (exact match), prefix search, and sort as unsigned bytes lexicographical compare. For sort, BytesRef's UTF8SortedAsUnicodeComparator accomplishes what we want, and even though the name of the comparator mentions UTF8, it doesn't actually assume so and just does byte-level operation, so it's good. However, when we do this across different nodes, we run into an issue where in QueryComponent.doFieldSortValues:</p> 
                  <p> // Must do the same conversion when sorting by a<br> // String field in Lucene, which returns the terms<br> // data as BytesRef:<br> if (val instanceof BytesRef) </p> { UnicodeUtil.UTF8toUTF16((BytesRef)val, spare); field.setStringValue(spare.toString()); val = ft.toObject(field); } 
                  <p>UnicodeUtil.UTF8toUTF16 is called on our byte array,which isn't actually UTF8. I did a hack where I specified our own field comparator to be ByteBuffer based to get around that instanceof check, but then the field value gets transformed into BYTEARR in JavaBinCodec, and when it's unmarshalled, it gets turned into byte[]. Then, in QueryComponent.mergeIds, a ShardFieldSortedHitQueue is constructed with ShardDoc.getCachedComparator, which decides to give me comparatorNatural in the else of the TODO for CUSTOM, which barfs because byte[] are not Comparable...</p> 
                  <p>From Chris Hostetter:</p> 
                  <p>I'm not very familiar with the distributed sorting code, but based on your<br> comments, and a quick skim of the functions you pointed to, it definitely<br> seems like there are two problems here for people trying to implement<br> custom sorting in custom FieldTypes...</p> 
                  <p>1) QueryComponent.doFieldSortValues - this definitely seems like it should<br> be based on the FieldType, not an "instanceof BytesRef" check (oddly: the<br> comment event suggestsion that it should be using the FieldType's<br> indexedToReadable() method – but it doesn't do that. If it did, then<br> this part of hte logic should work for you as long as your custom<br> FieldType implemented indexedToReadable in a sane way.</p> 
                  <p>2) QueryComponent.mergeIds - that TODO definitely looks like a gap that<br> needs filled. I'm guessing the sanest thing to do in the CUSTOM case<br> would be to ask the FieldComparatorSource (which should be coming from the<br> SortField that the custom FieldType produced) to create a FieldComparator<br> (via newComparator - the numHits &amp; sortPos could be anything) and then<br> wrap that up in a Comparator facade that delegates to<br> FieldComparator.compareValues</p> 
                  <p>That way a custom FieldType could be in complete control of the sort<br> comparisons (even when merging ids).</p> 
                  <p>...But as i said: i may be missing something, i'm not super familia with<br> that code. Please try it out and let us know if thta works – either way<br> please open a Jira pointing out the problems trying to implement<br> distributed sorting in a custom FieldType.</p> 
                 </div>
