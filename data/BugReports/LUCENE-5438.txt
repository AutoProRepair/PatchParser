                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>Lucene's replication module makes it easy to incrementally sync index<br> changes from a master index to any number of replicas, and it<br> handles/abstracts all the underlying complexity of holding a<br> time-expiring snapshot, finding which files need copying, syncing more<br> than one index (e.g., taxo + index), etc.</p> 
                  <p>But today you must first commit on the master, and then again the<br> replica's copied files are fsync'd, because the code operates on<br> commit points. But this isn't "technically" necessary, and it mixes<br> up durability and fast turnaround time.</p> 
                  <p>Long ago we added near-real-time readers to Lucene, for the same<br> reason: you shouldn't have to commit just to see the new index<br> changes.</p> 
                  <p>I think we should do the same for replication: allow the new segments<br> to be copied out to replica(s), and new NRT readers to be opened, to<br> fully decouple committing from visibility. This way apps can then<br> separately choose when to replicate (for freshness), and when to<br> commit (for durability).</p> 
                  <p>I think for some apps this could be a compelling alternative to the<br> "re-index all documents on each shard" approach that Solr Cloud /<br> ElasticSearch implement today, and it may also mean that the<br> transaction log can remain external to / above the cluster.</p> 
                 </div> 
                </div>
