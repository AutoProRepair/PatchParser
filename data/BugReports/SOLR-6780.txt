                <h2 class="toggle-title">Description</h2>
               </div>
               <div class="mod-content">
                <div id="description-val" class="field-ignore-highlight"> 
                 <div class="user-content-block"> 
                  <p>The <tt>DefaultSolrParams</tt> class, which is used as the basis for the implementation of <tt>defaults</tt>, <tt>appends</tt> and <tt>invariants</tt> params had a bug in it's implementation of <tt>getParameterNamesIterator()</tt> that could result in the same param key being returned multiple times.</p> 
                  <p>In many code paths of Solr, this bug had no effects – but in other cases, it resulted in code which iterated over the list of all parameters to take action multiple times for the (valid) key=value pairs.</p> 
                  <p>There were 4 main areas where this bug had unexpected &amp; problematic behavior for end users:</p> 
                  <div class="panel" style="border-width: 1px;">
                   <div class="panelHeader" style="border-bottom-width: 1px;">
                    <b>main problem areas &amp; impacts</b>
                   </div>
                   <div class="panelContent"> 
                    <ul> 
                     <li>ExtractingRequestHandler 
                      <ul> 
                       <li>"literal.*" params will be duplicated if overridden by defaults/invariants/appends - this will result in redundent literal field=value params being added to the document.</li> 
                       <li>impact: multiple values in literal fields when not expected/desired</li> 
                      </ul> </li> 
                     <li>FacetComponent 
                      <ul> 
                       <li>"facet.*" params will be duplicated if overridden by defaults/invariants/appends - this can result in redundent computation and identical facet.field, facet.query, or facet.range blocks in the response</li> 
                       <li>impact: wasted computation &amp; increased response size</li> 
                      </ul> </li> 
                     <li>SpellCheckComponent 
                      <ul> 
                       <li>when "custom params" (ie: "spellcheck.[dictionary name].XXXX=YYYY" are used in used in defaults, appends, or invariants, it can cause redudent XXXXX=YYYY params to be used.</li> 
                       <li>when "spellcheck.collateParam.XXXX=YYYY" type params are used defaults, appends, or invariants, it can cause redundent XXXX=YYYY params to exist in the collation verification queries.</li> 
                       <li>impact: unclear to me at first glance, probably just wasted computation &amp; increased response size</li> 
                      </ul> </li> 
                     <li>AnalyticsComponent 
                      <ul> 
                       <li>"olap.*" params will be duplicated if overridden by defaults/invariants/appends - this can result in redundent computation</li> 
                       <li>impact: unclear to me at first glance, probably just wasted computation &amp; increased response size</li> 
                      </ul> </li> 
                    </ul> 
                   </div>
                  </div> 
                  <p>Other less serious impacts were redundent values in "echoParams" as well as some small amounts of wasted computation in other code paths that iterated over the set of params (due to a slightly larger set of param values)</p> 
                  <div class="panel" style="border-width: 1px;">
                   <div class="panelHeader" style="border-bottom-width: 1px;">
                    <b>Original bug report: "Merging request parameters with defaults produce duplicate entries"</b>
                   </div>
                   <div class="panelContent"> 
                    <p>When a parameter (e.g. echoParams) is specified and overrides the default on the handler, it actually generates two entries for that key with the same value. </p> 
                    <p>Most of the time it is just a confusion and not an issue, however, some components will do the work twice. For example faceting component as described in <a href="http://search-lucene.com/m/QTPaSlFUQ1/duplicate" class="external-link" rel="nofollow">http://search-lucene.com/m/QTPaSlFUQ1/duplicate</a></p> 
                    <p>It may also be connected to <a href="https://issues.apache.org/jira/browse/SOLR-6369" title="duplicate json key &quot;echoParams&quot; should be removed in ping-response" class="issue-link" data-issue-key="SOLR-6369"><del>SOLR-6369</del></a></p> 
                    <p>The cause seems to be the interplay between <b>DefaultSolrParams#getParameterNamesIterator()</b> which just returns param names in sequence and <b>SolrParams#toNamedList()</b> which uses the first (override then default) value for each key, without deduplication.</p> 
                    <p>It's easily reproducible in trunk against schemaless example with </p> 
                    <blockquote>
                     <p>curl "http://localhost:8983/solr/schemaless/select?indent=true&amp;echoParams=all"</p>
                    </blockquote> 
                    <p>I've also spot checked it and it seems to be reproducible back to Solr 4.1.</p> 
                   </div>
                  </div> 
                 </div> 
                </div>
