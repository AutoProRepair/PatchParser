                 <div class="user-content-block"> 
                  <p>main issue was described in the mailing list here:<br> <a href="http://mail-archives.apache.org/mod_mbox/lucene-solr-user/201409.mbox/browser" class="external-link" rel="nofollow">http://mail-archives.apache.org/mod_mbox/lucene-solr-user/201409.mbox/browser</a><br> and <br> here:</p> 
                  <p>but also including the quotes:<br> original message from Nick</p> 
                  <blockquote> 
                   <p>Hello,<br> I have solr 4.10 running on tomcat 7. I'm doing replication from one master<br> to about 10 slaves, with standard configuration:</p> 
                   <div class="code panel" style="border-width: 1px;">
                    <div class="codeContent panelContent">

             &lt;str name=<span class="code-quote">"replicateAfter"</span>&gt;commit&lt;/str&gt;
             &lt;str name=<span class="code-quote">"replicateAfter"</span>&gt;startup&lt;/str&gt;

             &lt;!-- str name=<span class="code-quote">"commitReserveDuration"</span>&gt;00:00:10&lt;/str --&gt;

                  <p>My followup</p> 
                  <blockquote> 
                   <p>I was helping to look into this with Nick &amp; I think we may have figured out the core of the problem...</p> 
                   <p>The problem is easily reproducible by starting replication on the slave and then sending a shutdown command to tomcat (e.g. catalina.sh stop).</p> 
                   <p>With a debugger attached, it looks like the fsyncService thread is blocking VM shutdown because it is created as a non-daemon thread.</p> 
                   <p>Essentially what seems to be happening is that the fsyncService thread is running when 'catalina.sh stop' is executed. This goes in and calls SnapPuller.destroy() which aborts the current sync. Around line 517 of the SnapPuller, there is code that is supposed to cleanup the fsyncService thread, but I don't think it is getting executed because the thread that called SnapPuller.fetchLatestIndex() is configured as a daemon Thread, so the JVM ends up shutting that down before it can cleanup the fysncService...</p> 
                   <p>So... it seems like:</p> 
                   <div class="code panel" style="border-width: 1px;">
                    <div class="codeContent panelContent"> 
                     <pre class="code-java">    <span class="code-keyword">if</span> (fsyncService != <span class="code-keyword">null</span>) 
ExecutorUtil.shutdownNowAndAwaitTermination(fsyncService);
</pre> 
                    </div>
                   </div> 
                   <p>could be added around line 1706 of SnapPuller.java, or</p> 
                   <div class="code panel" style="border-width: 1px;">
                    <div class="codeContent panelContent"> 
                     <pre class="code-java">           <span class="code-comment">// since otherwise the JVM may kill the thread before any cleanup
</span>           <span class="code-comment">// code can be called
</span>          puller.setDaemon(<span class="code-keyword">false</span>);
</pre> 
                    </div>
                   </div> 
                   <p>could be added around line 230 of ReplicationHandler.java, however this needs some additional work (and I think it might need to be added regardless) since the cleanup code in SnapPuller(around 517) that shuts down the fsync thread never gets execute since logReplicationTimeAndConfFiles() can throw IO exceptions bypassing the rest of the finally block...So the call to </p> 
                   <div class="code panel" style="border-width: 1px;">
                    <div class="codeContent panelContent"> 
                     <pre class="code-java">logReplicationTimeAndConfFiles()</pre> 
                    </div>
                   </div> 
                   <p> around line 512 would need to get wrapped with a try/catch block to catch the IO exception...</p>
                  </blockquote> 
                  <p>I will attach a patch as well that I believe fixes the issues. Is there a reason to <b>not</b> perform a hard shutdown of the fysnc thread? It seems sane enough to me to do a hard shutdown of the fsyncService if the action is aborted...</p> 
                 </div>
